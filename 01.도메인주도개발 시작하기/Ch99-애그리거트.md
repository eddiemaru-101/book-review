## 1. 배경 — 왜 나왔나

Entity가 여러 개 생기면서 문제가 생겼다.

```
주문 확정할 때 같이 바뀌어야 하는 것들:
- Order.status       → CONFIRMED
- OrderItem.price    → 확정 금액
- ShippingInfo.addr  → 배송지 확정
```

애그리거트 없이 각각 따로 저장하면:

```
order.status = "CONFIRMED"
db.save(order)          # 저장 성공

order_item.price = 5000
db.save(order_item)     # 여기서 DB 에러 발생

# 결과:
# order.status = CONFIRMED (저장됨)
# order_item.price = None  (저장 안 됨)
# → 금액도 없는데 확정된 주문이 DB에 존재
```

비즈니스 규칙이 깨진 데이터가 DB에 저장된다. 이 문제를 해결하기 위해 나왔다.

---

## 2. 일관성이란

비즈니스 규칙이 항상 지켜지는 상태다.

```
주문 확정 시 반드시 지켜져야 하는 규칙:
- 주문 항목이 1개 이상 있어야 한다
- 모든 항목의 금액이 확정되어야 한다
- 배송지가 있어야 한다
- 세 조건이 동시에 만족될 때만 status가 CONFIRMED가 된다
```

---

## 3. 핵심 규칙

```
1. Root가 모든 규칙을 검증한다
2. 외부에서 내부 객체 직접 접근 불가      Root를 통해서만 접근
3. 저장/조회는 애그리거트 단위로 한다     OrderItem만 따로 저장 안 함
4. Root는 반드시 ID가 있다               전체 애그리거트의 식별자
```

---

## 4. 구조

```
╔══════════════════════════════════╗
║  Aggregate: Order                ║  ← 일관성/트랜잭션 경계
║                                  ║
║  ┌──────────┐                   ║
║  │  Order   │ ← Root            ║  모든 규칙 검증, 진입점
║  └────┬─────┘                   ║
║       │                          ║
║  ┌────▼──────┐  ┌─────────────┐ ║
║  │ OrderItem │  │ShippingInfo │ ║  항상 같이 저장/조회
║  └───────────┘  └─────────────┘ ║
╚══════════════════════════════════╝
```

## 5. 코드

```
# 내부 객체 — Root 통해서만 접근
class OrderItem:
    def __init__(self, product_id, quantity, price):
        self.product_id = product_id
        self.quantity = quantity
        self.price = price
```

```
# Root — 일관성 책임지는 진입점
class Order:
    def __init__(self, customer_id):
        self.order_id = str(uuid.uuid4())       # ← Root는 반드시 ID 있음
        self.customer_id = customer_id
        self._items = []                        # ← 언더스코어: 직접 접근 차단
        self._status = "PENDING"
        self._shipping_address = None

    def add_item(self, product_id, quantity, price):
        if self._status != "PENDING":
            raise Exception("이미 처리된 주문")     # ← 일관성 규칙 1
        if quantity <= 0:
            raise Exception("수량은 0보다 커야 함") # ← 일관성 규칙 2
        if price <= 0:
            raise Exception("금액은 0보다 커야 함") # ← 일관성 규칙 3
        self._items.append(OrderItem(product_id, quantity, price))

    def set_shipping_address(self, address):
        if self._status != "PENDING":
            raise Exception("이미 처리된 주문")     # ← 일관성 규칙 4
        self._shipping_address = address

    def confirm(self):
        if not self._items:
            raise Exception("주문 항목 없음")       # ← 일관성 규칙 5
        if not self._shipping_address:
            raise Exception("배송지 없음")          # ← 일관성 규칙 6
        self._status = "CONFIRMED"                  # ← 모든 규칙 통과 후 변경
        return OrderConfirmed(self.order_id)        # ← 이벤트 발행
```

```
# Repository — 트랜잭션을 하나로 묶음
class OrderRepository:
    def save(self, order: Order):
        with transaction():                         # ← 하나의 트랜잭션
            db.save(order)                          # Order 저장
            db.save(order._items)                   # OrderItem 저장
            db.save(order._shipping_address)        # ShippingInfo 저장
            # 셋 중 하나라도 실패하면 전체 롤백
```

```
# 외부에서 사용할 때
order = Order(customer_id="cust-001")

# 좋은 예 - Root를 통해서만 접근
order.add_item("product-001", 2, 5000)
order.set_shipping_address("서울시 강남구")
order.confirm()

# 나쁜 예 - 내부 객체 직접 접근 → 규칙 검증 완전히 무시
order._items.append(OrderItem(...))
order._status = "CONFIRMED"
```

## 6. Root 판별 기준

```
1. 외부에서 new 하는 클래스        Order를 new 하지 OrderItem을 new 안 함
2. ID가 있다                       self.order_id = str(uuid.uuid4())
3. 내부 객체를 소유한다            self._items, self._shipping_address
4. 비즈니스 규칙을 소유한다        confirm(), add_item() 안에 규칙 있음
```

## 7. 경계를 잘못 잡으면

```
# 나쁜 예 - 너무 크게 잡은 경우
╔════════════════════════════════════════════╗
║ Order + OrderItem + Inventory + Delivery   ║
╚════════════════════════════════════════════╝
- 주문 1건 저장할 때 재고, 배송 테이블까지 트랜잭션 묶임
- DB 락 범위가 넓어져서 동시 주문 처리 성능 박살
```

```
# 좋은 예 - 작게 잡은 경우
╔══════════════════╗  ╔═══════════════════╗  ╔══════════════════╗
║ Aggregate: Order ║  ║Aggregate:Inventory║  ║Aggregate:Delivery║
╚══════════════════╝  ╚═══════════════════╝  ╚══════════════════╝
- 각 트랜잭션이 독립적
- 동시 주문 100건 병렬 처리 가능
```

## 8. 하위도메인, 바운디드 컨텍스트와의 관계

```
하위도메인          비즈니스 관심사 경계    "주문이라는 비즈니스가 있다"
바운디드 컨텍스트   모델 경계              "주문 모델은 이 안에서만 유효하다"
애그리거트          일관성/트랜잭션 경계   "이 객체들은 항상 같이 변경된다"
```

```
shop/domain/
│
├── order/                      ← 하위도메인 = 바운디드 컨텍스트 (디렉터리)
│   ├── order_aggregate/        ← 일관성 경계 1
│   │   ├── order.py            ← Root
│   │   └── order_item.py       ← 내부 객체
│   │
│   └── cart_aggregate/         ← 일관성 경계 2 (같은 컨텍스트, 다른 트랜잭션)
│       ├── cart.py             ← Root
│       └── cart_item.py
│
├── inventory/                  ← 하위도메인 = 바운디드 컨텍스트
│   └── inventory_aggregate/    ← 일관성 경계 3
│       ├── inventory.py        ← Root
│       └── stock.py
│
└── delivery/                   ← 하위도메인 = 바운디드 컨텍스트
    └── delivery_aggregate/     ← 일관성 경계 4
        └── delivery.py         ← Root
```

바운디드 컨텍스트 1개 안에 애그리거트가 여러 개 있을 수 있다. 주문과 장바구니는 같은 비즈니스(order 컨텍스트)지만 트랜잭션이 독립적이라 애그리거트가 다르다.

**엔지니어 독백**

> 애그리거트를 처음 배울 때 "트랜잭션 묶는 거구나"로 이해하고 끝내는 경우가 많다. 근데 진짜 핵심은 일관성이다. 트랜잭션은 그 일관성을 기술적으로 보장하는 수단일 뿐이다.
> 
> 실무에서 일관성이 깨지는 버그는 찾기가 진짜 어렵다. 코드 에러가 아니라 데이터 상태가 이상한 거라서 로그에 안 잡힌다. "주문은 확정됐는데 재고가 안 줄었어요"라는 CS 문의가 들어오면 그때서야 발견한다.
> 
> 애그리거트 경계 잡는 기준은 단순하다. "이 두 객체가 항상 같이 변경되어야 비즈니스 규칙이 지켜지는가?" Yes면 같은 애그리거트, No면 다른 애그리거트다. 경계는 작게 잡아라. 크게 잡았다가 쪼개는 건 작게 잡았다가 합치는 것보다 훨씬 어렵다.
