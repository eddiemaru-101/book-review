# 01.배경
## 문제상황
큰 시스템을 하나의 도메인 모델로 표현하려다 문제가 생겼다.
"고객"이라는 단어 하나가 팀마다 다른 의미다.
- 주문팀: 고객 = 주문 이력이 있는 사람
- 마케팅팀: 고객 = 추천알고리즘 대상, 프로모션 대상, 멤버십 등급 gold인 사람
- 배송팀: 고객 = 배송받는 사람 

이걸 하나의 `Customer` 클래스로 표현하려 하면 필드가 수십 개 붙고, 팀마다 다른 규칙이 충돌한다. 모델이 오염된다.
- 정의
	- 바운디드 컨텍스트는 **같은 단어가 다른 의미로 쓰이는 경계를 명시적으로 구분한 것**
## 해결책 
- 코드적 관점에서의 경계임

같은 단어라도 컨텍스트(문맥)가 다르면 다른 모델로 정의한다.
```
┌─────────────────────────┐    ┌─────────────────────────┐
│   Order Context         │    │   Marketing Context     │
│                         │    │                         │
│   Customer              │    │   Customer              │
│   - customerId          │    │   - customerId          │
│   - shippingAddress     │    │   - segment             │
│   - orderHistory        │    │   - grade               │
│                         │    │   - purchasePattern     │
└─────────────────────────┘    └─────────────────────────┘
```

같은 "고객"이지만 각 컨텍스트 안에서 완전히 다른 클래스다.


## 하위도메인 vs 바운디드 컨텍스트 

> 하위도메인과 바운디드컨텍스트가 뭐가 다른지 헷갈리네


- 하위도메인: "비즈니스를 어떻게 나눌 것인가" → 현실 세계 기준
- 바운디드 컨텍스트: "코드를 어떻게 나눌 것인가" → 구현 세계 기준

하위도메인은 비즈니스 분석 단계에서 나온다.
```python
쿠팡 비즈니스
├── 주문 (Core Domain)
├── 재고 (Supporting Domain)
├── 배송 (Core Domain)
├── 결제 (Generic Domain)
└── 알림 (Generic Domain)
```

바운디드 컨텍스트는 실제 코드/팀 경계다.
```python
쿠팡 코드
├── Order Service       ← 주문 컨텍스트
├── Inventory Service   ← 재고 컨텍스트
├── Delivery Service    ← 배송 컨텍스트
├── Payment Service     ← 결제 컨텍스트
└── Notification Service← 알림 컨텍스트
```

**정리**
```python
하위도메인          바운디드 컨텍스트
────────────────────────────────────
비즈니스 분석       코드 설계
현실 세계           구현 세계
기획자가 정의       개발자가 정의
변하지 않음         팀 상황따라 유동적
```


## 하위도메인과 대응관계 

> 둘의 관계는? 독립적이다. 하나의 하위도메인에 복수개의 바운디드 컨텍스트 존재 가능 두개의 하위도메인에 걸쳐 하나의 바운디드 컨텍스트 존재 가능


이상적으론 하위도메인 1개 = 바운디드 컨텍스트 1개다. 근데 현실은 다르다.

1. 하위도메인 1개를 바운디드 컨텍스트 2개로 쪼갠 경우

```python
주문 (하위도메인 1개)
├── 주문접수 컨텍스트   ← 주문 생성, 확정
└── 주문이력 컨텍스트   ← 조회, 통계, 분석용
```

주문 접수와 주문 조회는 같은 비즈니스지만, 읽기/쓰기 모델을 분리할 때 컨텍스트를 나눈다.

1. 바운디드 컨텍스트 1개가 하위도메인 2개를 담은 경우

```python
회원 컨텍스트 (바운디드 컨텍스트 1개)
├── 인증 (하위도메인)
└── 프로필 (하위도메인)
```

팀이 작을 때 굳이 두 서비스로 나누지 않고 하나로 합친다.



## 어느영역 레벨인가? 인프라스트런처와 연관성 있나?
도메인 모델 레벨이다.


**이유**
바운디드 컨텍스트는 "어떤 클래스를 어떤 경계 안에 둘 것인가"를 정의하는 것이다. `Customer` 클래스 자체가 도메인 레이어에 있는 Entity다.
```
domain/
├── order/
│   └── customer.py     ← 주문 컨텍스트의 Customer
├── marketing/
│   └── customer.py     ← 마케팅 컨텍스트의 Customer
└── cs/
    └── customer.py     ← CS 컨텍스트의 Customer
```



**인프라스트럭처 레벨과 구분**
```
도메인 레벨 문제          인프라 레벨 문제
──────────────────────────────────────────
Customer 클래스 설계      Customer 테이블 설계
비즈니스 규칙 경계        DB 연결, ORM 매핑
모델 분리 기준            저장소 구현
```
인프라는 도메인 모델을 DB에 어떻게 저장할지를 다룬다. 바운디드 컨텍스트는 그 전 단계인 모델 자체를 어떻게 나눌지를 다룬다.




# 02. 구현
## (1) 바운디드 컨텍스트 필요성
### 하나의 모델로 구현하면? 
단일 모델의 문제는 **"고객"이라는 하나의 클래스가 모든 컨텍스트의 요구를 다 담으려다 망가지는 것**이다.


**상황 설정**
쿠팡에서 `Customer` 클래스 하나로 전체 시스템을 운영한다고 가정한다.

**초기 코드 — 처음엔 단순해 보인다**
```
class Customer:
    def __init__(self, customer_id, name, email):
        self.customer_id = customer_id
        self.name = name
        self.email = email
```



**주문팀 요구사항 추가**
"배송지 주소가 필요합니다"
```
class Customer:
    def __init__(self, customer_id, name, email):
        self.customer_id = customer_id
        self.name = name
        self.email = email
        self.shipping_address = None    # 주문팀 요구
        self.order_history = []         # 주문팀 요구
```



**마케팅팀 요구사항 추가**
"고객 등급이랑 구매 패턴 분석이 필요합니다"
```
class Customer:
    def __init__(self, customer_id, name, email):
        self.customer_id = customer_id
        self.name = name
        self.email = email
        self.shipping_address = None    # 주문팀
        self.order_history = []         # 주문팀
        self.grade = "NORMAL"           # 마케팅팀 요구
        self.segment = None             # 마케팅팀 요구
        self.purchase_pattern = {}      # 마케팅팀 요구
```



**CS팀 요구사항 추가**
"고객 문의 이력이랑 처리 상태가 필요합니다"
```
class Customer:
    def __init__(self, customer_id, name, email):
        self.customer_id = customer_id
        self.name = name
        self.email = email
        self.shipping_address = None    # 주문팀
        self.order_history = []         # 주문팀
        self.grade = "NORMAL"           # 마케팅팀
        self.segment = None             # 마케팅팀
        self.purchase_pattern = {}      # 마케팅팀
        self.inquiry_history = []       # CS팀 요구
        self.cs_status = None           # CS팀 요구

    # 주문팀 메서드
    def add_order(self, order):
        self.order_history.append(order)

    # 마케팅팀 메서드
    def upgrade_grade(self):
        if len(self.order_history) > 10:    # 주문팀 데이터를 마케팅팀 메서드가 참조
            self.grade = "VIP"

    # CS팀 메서드
    def add_inquiry(self, inquiry):
        self.inquiry_history.append(inquiry)
        if len(self.order_history) == 0:    # CS팀 메서드가 주문팀 데이터를 참조
            self.cs_status = "NO_ORDER_CUSTOMER"
```



**문제가 터지는 순간**
주문팀이 `order_history` 구조를 바꾸면:
```
# 주문팀: order_history를 dict → Order 객체로 변경
self.order_history = []  # Order 객체 리스트로 바꿨다

# 마케팅팀 코드 깨짐
def upgrade_grade(self):
    if len(self.order_history) > 10:  # 여기는 아직 dict 기준으로 짜여있음
        self.grade = "VIP"            # 런타임 에러 발생

# CS팀 코드도 깨짐
def add_inquiry(self, inquiry):
    if len(self.order_history) == 0:  # 여기도 깨짐
        self.cs_status = "NO_ORDER_CUSTOMER"
```



**핵심 문제 정리**
```
Customer 클래스 하나가
├── 주문팀 데이터/규칙
├── 마케팅팀 데이터/규칙
└── CS팀 데이터/규칙
을 전부 들고 있다

→ 주문팀이 뭔가 바꾸면 마케팅팀, CS팀 코드가 깨진다
→ 아무도 Customer 클래스를 쉽게 못 건드린다
→ 기능 확장이 불가능해진다
```
바운디드 컨텍스트로 나누면 각 팀의 `Customer`는 독립적이라 한 팀이 변경해도 다른 팀 코드가 깨지지 않는다.


**엔지니어 독백**
> 이 상황을 실무에서 "God Object(신 객체)"라고 부른다. 모든 걸 아는 클래스 하나가 시스템 전체를 쥐고 있는 거다.
> 
> 처음엔 "어차피 고객 데이터인데 왜 나눠요?"라고 한다. 근데 6개월 지나면 Customer 클래스가 필드 50개짜리 괴물이 되고, 아무도 건드리기 무서워서 그냥 필드 하나 더 추가하는 식으로 때운다. 그게 쌓이면 레거시가 된다.
> 
> 실무에서 이 냄새가 나기 시작하는 신호는 단순하다. PR 리뷰할 때 "이거 건드리면 다른 팀 코드 깨지지 않아요?"라는 말이 나오면 그때 이미 늦은 거다.



## (2) 컨텍스트 간 통신

경계가 생기면 컨텍스트끼리 직접 의존하지 않는다. 이벤트로 통신한다.

```
┌──────────────────┐          ┌──────────────────┐
│  Order Context   │          │ Inventory Context│
│                  │          │                  │
│  order.confirm() │──────▶   │ inventory        │
│                  │ OrderConfirmed    .deduct()  │
│                  │  (Event) │                   │
└──────────────────┘          └──────────────────┘
```

Order Context는 Inventory Context의 내부 구조를 모른다. 이벤트를 던질 뿐이다.

---

**엔지니어 독백**

> 신입 때 가장 많이 하는 실수가 "어차피 같은 DB 쓰는데 왜 굳이 나눠요?"다.
> 
> 처음엔 테이블 조인 한 방으로 다 해결되는 것처럼 보인다. 근데 팀이 5명에서 50명 되는 순간, 주문팀이 고객 테이블 컬럼 하나 바꾸면 마케팅팀 쿼리가 깨진다. 서로 눈치 보면서 스키마 못 건드리는 상황이 온다.
> 
> 바운디드 컨텍스트는 기술적 결정이 아니라 팀 조직 구조를 코드에 반영하는 거다. 콘웨이 법칙 그대로다. 팀 경계가 곧 컨텍스트 경계다.
> 
> 처음 설계할 때 경계를 너무 잘게 쪼개려 하지 마라. 모호하면 일단 합쳐두고, 팀이 "이 부분 변경할 때 왜 저 팀 눈치를 봐야 해?"라는 말이 나오기 시작할 때 그때 경계를 긋는 게 현실적이다.


## (3) 실질적 구현방법은?

**컨텍스트별로 같은 개념이라도 클래스를 따로 만든다.**


**구현 방법 3가지**
- 1. 클래스 이름을 컨텍스트명으로 구분
- 2. 디렉터리(패키지)로 구분
- 3. 클래스 이름은 같되 디렉터리로 구분 ← 실무 선호

실무에서는 3번을 쓴다. 디렉터리가 곧 컨텍스트 경계이기 때문에 클래스 이름은 `Customer`로 동일하게 유지한다.



**디렉터리 구조**
```
shop/
└── domain/
    ├── order/
    │   └── order_customer.py       ← OrderCustomer
    ├── marketing/
    │   └── marketing_customer.py   ← MarketingCustomer
    └── cs/
        └── cs_customer.py          ← CSCustomer
```



**코드**
```
# domain/order/order_customer.py 
class OrderCustomer:                       # 주문 컨텍스트의 Customer
    def __init__(self, customer_id, name, shipping_address):
        self.customer_id = customer_id
        self.name = name
        self.shipping_address = shipping_address    # 주문팀만 필요한 필드
        self.order_history = []

    def add_order(self, order):
        if not self.shipping_address:
            raise Exception("배송지 주소가 없습니다")   # 주문 컨텍스트 규칙
        self.order_history.append(order)
```

```
# domain/marketing/marketing_customer.py 
class MarketingCustomer:                        # 마케팅 컨텍스트의 Customer
    def __init__(self, customer_id, grade, segment):
        self.customer_id = customer_id
        self.grade = grade                          # 마케팅팀만 필요한 필드
        self.segment = segment
        self.purchase_pattern = {}

    def upgrade_grade(self, order_count):
        if order_count > 10:
            self.grade = "VIP"                      # 마케팅 컨텍스트 규칙
```

```
# domain/cs/cs_customer.py 
class CSCustomer:                         # CS 컨텍스트의 Customer
    def __init__(self, customer_id, cs_status):
        self.customer_id = customer_id
        self.cs_status = cs_status                  # CS팀만 필요한 필드
        self.inquiry_history = []

    def add_inquiry(self, inquiry):
        self.inquiry_history.append(inquiry)
        if len(self.inquiry_history) > 5:
            self.cs_status = "FREQUENT_INQUIRER"    # CS 컨텍스트 규칙
```



**사용할 때 import 경로로 컨텍스트가 구분된다**
```
from domain.order.order_customer import OrderCustomer from domain.marketing.marketing_customer import MarketingCustomer from domain.cs.cs_customer import CSCustomer
```



**핵심**
`customer_id` 하나만 공유한다. 나머지는 각 컨텍스트가 독립적으로 관리한다.
```
공유하는 것       공유하지 않는 것
─────────────────────────────────
customer_id      필드
                 비즈니스 규칙
                 메서드
                 상태
```

**엔지니어 독백**
> 처음 보면 "Customer 클래스가 3개? 중복 아니에요?"라고 한다. 중복이 아니다. 각자 다른 비즈니스 규칙을 담고 있는 독립된 모델이다.
> 
> 실무에서 이게 진짜 빛을 발하는 순간은 마케팅팀이 `grade` 로직을 바꿀 때다. `domain/marketing/customer.py`만 건드리면 끝이다. 주문팀 코드는 건드릴 필요가 없다. 이게 바운디드 컨텍스트의 전부다.


**컨텍스트 간 통신할 때도 id만 넘긴다**
주문이 확정되면 마케팅팀이 등급을 올려야 하는 상황:
```
# 나쁜 예 - 컨텍스트 간 객체를 직접 넘김
order_customer = OrderCustomer(...)
marketing_service.upgrade(order_customer)   # OrderCustomer를 마케팅팀이 직접 참조
```

```
# 좋은 예 - id만 넘김
event = OrderConfirmed(customer_id="cust-001", order_count=11)
marketing_service.upgrade(event.customer_id)    # id만 받아서 마케팅 컨텍스트가 자기 모델로 처리
```

**핵심**
```
공유하는 것       공유하지 않는 것
─────────────────────────────────
customer_id      클래스 구조
                 필드
                 비즈니스 규칙
                 메서드
```
컨텍스트끼리 객체를 직접 주고받는 순간 결합도가 생긴다. id만 넘기고 각자 자기 모델로 처리하는 게 원칙이다.










# 03. 하위도메인, 바운디드 컨텍스트, aggregate

## (1)셋의 관계 


세 개는 **각각 다른 레벨에서 "경계"를 정의하는 개념**이다.

**한 줄 정리**
```
하위도메인        비즈니스를 어떻게 나눌 것인가     (비즈니스 레벨)
바운디드 컨텍스트  코드/모델을 어떻게 나눌 것인가    (설계 레벨)
애그리거트        객체 일관성을 어떻게 보장할 것인가  (구현 레벨)
```



**포함 관계**
```
하위도메인 (비즈니스 경계)
└── 바운디드 컨텍스트 (모델 경계)
    └── 애그리거트 (객체 일관성 경계)
        └── Entity / Value Object
```



**쿠팡 예시로 전체 그림**
```
쿠팡 도메인
│
├── 주문 하위도메인
│   └── Order 바운디드 컨텍스트
│       └── Order 애그리거트
│           ├── Order (Root Entity)
│           └── OrderItem (Entity)
│
├── 재고 하위도메인
│   └── Inventory 바운디드 컨텍스트
│       └── Inventory 애그리거트
│           ├── Inventory (Root Entity)
│           └── Stock (Value Object)
│
└── 배송 하위도메인
    └── Delivery 바운디드 컨텍스트
        └── Delivery 애그리거트
            └── Delivery (Root Entity)
```



**각 경계가 막는 것**
```
경계                  막는 것
────────────────────────────────────────────────
하위도메인            비즈니스 관심사 침범
바운디드 컨텍스트     다른 컨텍스트 모델 직접 참조
애그리거트            Root bypss하고 내부 객체 직접 접근
```



**엔지니어 독백**
> 이 세 개를 처음 배울 때 가장 많이 하는 실수가 "다 같은 말 아니에요?"다.
> 
> 쉽게 기억하는 방법이 있다. 줌인/줌아웃으로 생각해라. 하늘에서 내려다보면 하위도메인이 보이고, 더 내려오면 바운디드 컨텍스트가 보이고, 코드 레벨까지 내려오면 애그리거트가 보인다.
> 
> 실무에서 설계 회의할 때 이 세 레벨을 섞어서 얘기하면 대화가 꼬인다. "지금 비즈니스 경계 얘기해요, 모델 경계 얘기해요?"를 먼저 구분하고 대화하는 습관을 들여라.


## (2)둘의관계 : 바운디드 컨텍스트, 애그리거트는 1:1이 아닌가?

> 근데 애그리거트, 바운디드 컨텍스트는 좀 연관성이 있지않나? 애그리거트 root를 기준으로접근해야하면 두개가 1:1 대응이 되어야 하는거아닌가? 아니면 애그리거트가 무조건 바운디드 컨텍스트 클래스에 속해야하거나



좋은 질문이다. **1:1이 아니다. 바운디드 컨텍스트 안에 애그리거트가 여러 개 있을 수 있다.**


**관계**
```
바운디드 컨텍스트 1개
└── 애그리거트 N개
```



**쿠팡 주문 컨텍스트 예시**
```
Order 바운디드 컨텍스트
│
├── Order 애그리거트          ← 주문 일관성 단위
│   ├── Order (Root)
│   └── OrderItem
│
└── Cart 애그리거트           ← 장바구니 일관성 단위
    ├── Cart (Root)
    └── CartItem
```

주문과 장바구니는 같은 주문 컨텍스트 안에 있지만 트랜잭션 경계가 다르다. 장바구니 변경이 주문에 영향을 주지 않는다.



**구분 기준**
```
바운디드 컨텍스트 경계     같은 비즈니스 관심사인가
애그리거트 경계            함께 변경되어야 하는가
```
장바구니와 주문은 같은 비즈니스 관심사(주문 컨텍스트)지만, 함께 변경될 필요가 없으므로 애그리거트가 다르다.



**엔지니어 독백**
> 처음 설계할 때 "바운디드 컨텍스트 = 애그리거트"로 1:1 매핑하려는 실수를 많이 한다. 그러면 컨텍스트가 너무 잘게 쪼개진다.
> 
> 기준은 단순하다. "이 두 객체가 항상 같이 저장되어야 하나?" → Yes면 같은 애그리거트. "이 두 기능이 같은 비즈니스 관심사인가?" → Yes면 같은 바운디드 컨텍스트.


## (3) 바운디드 컨텍스트, 애그리거트 구현 
**디렉터리가 바운디드 컨텍스트, 클래스 묶음이 애그리거트, Root가 진입점이다.**



**디렉터리 구조**
```
shop/
└── domain/
    └── order/                      ← 바운디드 컨텍스트 경계
        ├── order_aggregate/        ← Order 애그리거트
        │   ├── order.py            ← Root
        │   └── order_item.py       ← 내부 객체
        │
        └── cart_aggregate/         ← Cart 애그리거트
            ├── cart.py             ← Root
            └── cart_item.py        ← 내부 객체
```



**코드**
```
# order_aggregate/order_item.py
class OrderItem:                            # 애그리거트 내부 객체
    def __init__(self, product_id, quantity, price):
        self.product_id = product_id
        self.quantity = quantity
        self.price = price
```

```
# order_aggregate/order.py
from order_item import OrderItem

class Order:                                # ← 애그리거트 Root
    def __init__(self, customer_id):
        self.order_id = str(uuid.uuid4())   # ← Root는 반드시 ID가 있다
        self.customer_id = customer_id
        self._items = []                    # ← 내부 객체, 외부 직접 접근 차단
        self._status = "PENDING"

    # 외부에서 OrderItem에 접근하는 유일한 통로
    def add_item(self, product_id, quantity, price):
        if self._status != "PENDING":
            raise Exception("이미 처리된 주문입니다")   # ← 비즈니스 규칙
        self._items.append(OrderItem(product_id, quantity, price))

    def confirm(self):
        if not self._items:
            raise Exception("주문 항목이 없습니다")     # ← 비즈니스 규칙
        self._status = "CONFIRMED"
        return OrderConfirmed(self.order_id)
```

```
# 외부에서 사용할 때
order = Order(customer_id="cust-001")

# 좋은 예 - Root를 통해서만 접근
order.add_item("product-001", 2, 5000)
order.confirm()

# 나쁜 예 - 내부 객체 직접 접근
order._items.append(OrderItem(...))     # Root의 규칙 검증을 완전히 무시
```



**어디가 뭔지 주석으로 표시**
```
shop/domain/order/          ← 바운디드 컨텍스트 (디렉터리)
│
├── order_aggregate/        ← 애그리거트 (묶음)
│   ├── order.py            ← Root (ID 있음, 진입점, 규칙 소유)
│   └── order_item.py       ← 내부 객체 (Root 통해서만 접근)
│
└── cart_aggregate/         ← 다른 애그리거트 (같은 컨텍스트)
    ├── cart.py             ← Root
    └── cart_item.py        ← 내부 객체
```



**Root 판별 기준 3가지**
```
1. ID가 있다          self.order_id = str(uuid.uuid4())
2. 내부 객체를 소유   self._items = []
3. 비즈니스 규칙 소유 add_item(), confirm() 안에 규칙 있음
```





**엔지니어 독백**

> Root를 이해하는 가장 빠른 방법은 "외부에서 이 애그리거트를 쓸 때 뭘 new 하나?"를 생각하는 거다. `Order`를 new 하지 `OrderItem`을 new 하지 않는다. 외부에서 직접 new 하는 클래스가 Root다.
> 
> `_items`에 언더스코어 붙인 거 보이지? "직접 건드리지 마세요"라는 팀 내 약속이다. 실무에서 이 약속이 깨지는 순간 비즈니스 규칙 검증이 통째로 무시된다. 코드 리뷰할 때 `._items` 직접 접근하는 코드 보이면 바로 리젝이다.
