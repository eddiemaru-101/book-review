# 1.1 도메인이란

## (1) 도메인이란

- 도메인 : 소프트웨어가 문제를 해결하려는 비즈니스 문제 영역 범위

**핵심 정의**

기술적 구현이 아니라, 비즈니스 전문가(도메인 전문가)가 실제로 다루는 **개념**, **규칙**, **프로세스**의 집합.

예를 들어 쇼핑몰을 만든다고 하면:

- 주문, 결제, 배송, 반품 — 이게 도메인
- Django, PostgreSQL, Redis — 이건 도메인이 아닌 기술 스택

**왜 이 개념이 나왔나**

기존 개발 방식의 한계:

1. 개발자가 비즈니스 로직을 DB 스키마나 API 구조 중심으로 설계
2. 비즈니스가 복잡해질수록 코드가 비즈니스 언어와 멀어짐
3. 기획자/개발자 간 소통 비용 폭증, 요구사항 오해 잦아짐

**DDD가 나온 이유:**

- 비즈니스 전문가가 쓰는 언어 그대로 코드에 반영하자 (유비쿼터스 언어)
- 비즈니스 규칙이 코드 안에 명확히 표현되게 하자

## (2) 하위도메인

하위 도메인(Subdomain)은 **큰 도메인을 기능 단위로 쪼갠 것**이다.

**왜 나눠야 하나**

쇼핑몰 전체가 하나의 도메인이지만, 실제로는 팀도 다르고 규칙도 다르고 전문가도 다르다. 한 덩어리로 보면 너무 복잡해서 나눈다.

**3가지 종류**

1. **Core Domain (핵심 도메인)**
    - 회사가 경쟁력을 갖는 핵심 비즈니스
    - 개발 리소스를 가장 많이 투자해야 하는 곳
    - 예: 쿠팡 → 로켓배송 배차 알고리즘
2. **Supporting Domain (지원 도메인)**
    - 핵심을 돕지만 핵심은 아님
    - 직접 만들긴 해야 함
    - 예: 쿠팡 → 상품 리뷰 관리
3. **Generic Domain (범용 도메인)**
    - 비즈니스와 무관하게 어디서나 쓰는 것
    - 보통 외부 솔루션으로 대체
    - 예: 쿠팡 → 이메일 발송, 결제(PG사)

**실무에서 왜 중요하냐**

어디에 인력과 설계 품질을 집중할지 결정하는 기준이 된다. Core는 직접 정교하게 설계, Generic은 그냥 외부 서비스 붙이면 끝.



# 1.2 도메인 전문가와 개발자간 지식 공유

## (1)개발자 - 도메인 대화 필요한 이유
- 요구사항 구체화 하기 위해
- 요구사항 기반으로 설계하기 때문
- 요구사항 분석 부정확 → 엉뚱한 설계, 엉뚱한 기능 → 코드수정 반복
- 문서로 적혀진 요구사항과 실제 요청한 사람의 원하는 기능이 다를 수 있음
	- 대화를 통해 진짜 요구사항이 뭔지 파악해야 함

## (2)관계흐름
- 비즈니스 요구사항이 들어온다
- 그 안에서 도메인 전문가의 언어를 추출한다
- 그 언어로 도메인 모델을 만든다

## (3)예시
요구사항: "고객이 상품을 주문하면 재고를 차감하고 배송을 시작한다"

여기서 추출되는 것들:
- 도메인 개념: 고객, 상품, 주문, 재고, 배송
- 도메인 규칙: 주문 시 재고 차감, 주문 후 배송 시작
- 하위 도메인 경계: 주문/재고/배송이 각각 다른 관심사

## (4) 핵심
> 요구사항이 바뀌면 도메인 모델이 바뀌고, 코드가 바뀐다. 이 순서가 깨지면 DDD가 아니다.
- DDD는 요구사항 → 도메인 모델 → 기술 구현 순서를 지킨다.



# 1.3 도메인 모델
## (1)도메인 모델이란
- 도메인 모델은 비즈니스 개념과 규칙을 코드로 표현한 것
## (2) 왜나왔나

### 기존 방식(트랜잭션 스크립트):

- 비즈니스 로직이 서비스 레이어에 절차적으로 나열됨
- 비즈니스가 복잡해질수록 코드가 스파게티가 됨
- 코드만 봐서는 비즈니스 규칙을 파악할 수 없음

> 잘 모르겠어. 예시를 들어줘 

기존 방식(트랜잭션 스크립트)은 
**비즈니스 로직이 서비스 함수 안에 절차적으로 쭉 나열된 것**이다.


**주문 처리 예시**
요구사항: "고객이 주문하면 재고 차감하고 배송 시작"

기존 방식:
```
def place_order(customer_id, product_id, quantity):
    customer = db.query("SELECT * FROM customers WHERE id = ?", \
					    customer_id)
    product = db.query("SELECT * FROM products WHERE id = ?", product_id)
    
    # 재고 확인
    if product.stock < quantity:
        raise Exception("재고 부족")
    
    # 금액 계산
    total = product.price * quantity
    
    # 주문 생성
    db.execute("INSERT INTO orders ...", customer_id, product_id, total)
    
    # 재고 차감
    db.execute("UPDATE products SET stock = stock - ? WHERE id = ?", \
				quantity, product_id)
    
    # 배송 시작
    db.execute("INSERT INTO deliveries ...", order_id)
```


**문제가 뭐냐**
지금은 단순해 보이지만 비즈니스가 복잡해지면:
1. 할인 정책 추가 → 이 함수에 if문 추가
2. VIP 고객 처리 → 이 함수에 if문 추가
3. 부분 배송 처리 → 이 함수에 if문 추가
4. 결국 함수 하나가 300줄짜리 괴물이 됨
그리고 결정적으로:
- `place_order` 함수를 읽어도 **비즈니스 규칙이 보이지 않는다**
- SQL과 비즈니스 로직이 뒤섞여 있다
- 재고 차감 규칙이 바뀌면 **이 함수를 찾아서 고쳐야 한다는 걸 어떻게 아나**

**DDD 방식과 한 줄 비교**
- 기존: 함수가 모든 걸 안다, 객체는 데이터 덩어리
- DDD: 객체가 자기 규칙을 안다, 함수는 흐름만 조율

**DDD 예시**
- 비즈니스 규칙이 바뀌면 `Order` 클래스만 보면 된다. 찾아다닐 필요가 없다.
~~~
# Value Object
class Money:
    def __init__(self, amount):
        self.amount = amount

# Entity
class OrderItem:
    def __init__(self, product_id, quantity, price: Money):
        self.product_id = product_id
        self.quantity = quantity
        self.price = price

# Aggregate Root
class Order:
    def __init__(self, customer_id):
        self.id = OrderId.generate()
        self.customer_id = customer_id
        self.items = []
        self.status = "PENDING"

    def add_item(self, product_id, quantity, price: Money):
        if self.status != "PENDING":
            raise Exception("이미 처리된 주문입니다")      # ← 비즈니스 규칙이 객체 안에
        self.items.append(OrderItem(product_id, quantity, price))

    def place(self):
        if not self.items:
            raise Exception("주문 항목이 없습니다")         # ← 비즈니스 규칙이 객체 안에
        self.status = "PLACED"
        return OrderPlaced(self.id)                        # ← Domain Event 발행

# Service (흐름만 조율)
class OrderService:
    def place_order(self, customer_id, product_id, quantity):
        product = self.product_repo.find(product_id)
        order = Order(customer_id)
        order.add_item(product_id, quantity, product.price)
        event = order.place()                              # ← 서비스는 흐름만
        self.order_repo.save(order)
        self.event_bus.publish(event)
~~~
### 도메인 모델이 나온 이유:
- 비즈니스 규칙을 객체 안에 캡슐화
- 코드가 곧 비즈니스 문서가 되게 하자



## (3) 도메인 모델의 구성요소
1. **Entity** — 고유 식별자가 있는 객체 (주문, 고객)
2. **Value Object** — 식별자 없이 값 자체가 의미인 객체 (금액, 주소)
3. **Aggregate** — Entity/VO의 묶음, 일관성 경계
4. **Domain Service** — 객체에 속하기 애매한 비즈니스 로직
5. **Repository** — Aggregate를 저장/조회하는 인터페이스
6. **Domain Event** — 도메인에서 발생한 사건

**핵심**
> 도메인 모델은 DB 설계가 아니다. 
> 테이블 구조가 아니라 비즈니스 규칙이 중심이다. 
> DB는 도메인 모델을 저장하는 수단일 뿐이다.


## (3)도메인 모델 표현 방식 
> 도메인 모델은 클래스 다이어그램, 상태 다이어그램, 수학공식 등 다양하게 표현할 수 있음.
> 표현 방식보단 내용에 집중  

**1. 클래스 다이어그램 형식**
- 객체가 어떤 데이터와 메서드를 갖는지
- 객체 간 포함 관계
- 비즈니스 규칙이 어느 객체에 있는지
```
┌─────────────────────┐
│       Order         │  ← Entity (Aggregate Root)
├─────────────────────┤
│ - id: OrderId       │  ← Value Object
│ - status: OrderStatus│
│ - items: List[OrderItem]│
├─────────────────────┤
│ + place()           │  ← 비즈니스 규칙 메서드
│ + cancel()          │
│ + total_price()     │
└─────────────────────┘
         │ contains
         ▼
┌─────────────────────┐
│      OrderItem      │  ← Entity
├─────────────────────┤
│ - product_id        │
│ - quantity          │
│ - price: Money      │  ← Value Object
└─────────────────────┘
```

---

**2. 바운디드 컨텍스트 + 관계 형식**
- 어떤 컨텍스트(서비스 경계)가 존재하는지
- 컨텍스트 간 어떤 이벤트로 통신하는지
- MSA 서비스 분리 기준을 한눈에 파악

```
┌──────────────────────────────┐
│       Order Context          │
│  ┌────────┐   ┌───────────┐  │
│  │ Order  │──▶│ OrderItem │  │
│  └────────┘   └───────────┘  │
└──────────────────────────────┘
         │ Domain Event
         ▼
┌──────────────────────────────┐
│      Delivery Context        │
│  ┌──────────────────────┐    │
│  │      Delivery        │    │
│  └──────────────────────┘    │
└──────────────────────────────┘
```

---

**3. Aggregate 경계 강조 형식**
- Aggregate 경계가 어디까지인지
- 어떤 객체가 Root인지 (외부 접근 진입점)
- 일관성을 보장해야 하는 범위가 어디인지

```
╔══════════════════════════════╗
║   Aggregate: Order           ║  ← 경계
║  ┌──────────┐                ║
║  │  Order   │ ← Root         ║
║  └────┬─────┘                ║
║       │                      ║
║  ┌────▼─────┐                ║
║  │OrderItem │                ║
║  └──────────┘                ║
╚══════════════════════════════╝
```

**4.하위도메인**
~~~
┌─────────────────────────────────────────────────────┐
│                   쇼핑몰 Domain                      │
│                                                     │
│  ╔═══════════════════╗                              │
│  ║  Core Domain      ║                              │
│  ║  [ 주문 ]         ║──── OrderPlaced ────▶        │
│  ╚═══════════════════╝                              │
│                                                     │
│  ┌───────────────────┐                              │
│  │ Supporting Domain │◀──── OrderPlaced ────        │
│  │ [ 재고 ]          │                              │
│  └───────────────────┘                              │
│                                                     │
│  ╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌                              │
│  Generic Domain                                     │
│  [ 알림 / 결제 ]  → 외부 서비스로 대체               │
│  ╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌╌                              │
└─────────────────────────────────────────────────────┘
~~~

- `╔══╗` 이중선 = Core (가장 중요)
- `┌──┐` 단선 = Supporting
- `╌╌╌` 점선 = Generic (외부 대체)

# 1.4 도메인 모델 패턴 

## (1) App아키텍처와 도메인 모델

- 도메인 모델 - 도메인 자체를 이해하기 위한 개념적인 모델

> 코드 아키텍처에서 도메인 모델은 어떻게 표현될까?

### 레이어드 아키텍처(Layered Architecture)
- DDD를 코드로 구현할 때 쓰는 폴더/계층 구조
- 도메인 모델을 어느 레이어에 놓을지 정의되어 있음
~~~
┌─────────────────────────────────┐
│  표현 (Presentation)            │  ← 사용자 요청 받는 곳
│  API 엔드포인트, Controller     │    (FastAPI, Django View)
└────────────────┬────────────────┘
                 │
┌────────────────▼────────────────┐
│  응용 (Application)             │  ← 흐름만 조율
│  UseCase, Service               │    (비즈니스 규칙 없음)
└────────────────┬────────────────┘
                 │
┌────────────────▼────────────────┐
│  도메인 (Domain)                │  ← 비즈니스 규칙 여기 있음
│  Entity, VO, Aggregate          │    (핵심)
│  Domain Service, Domain Event   │
└────────────────┬────────────────┘
                 │
┌────────────────▼────────────────┐
│  인프라스트럭처 (Infrastructure)    │  ← DB, 외부 API 연결
│  Repository 구현체, ORM           │
└─────────────────────────────────┘
~~~

- **각 레이어 한 줄 정의**
	- 표현: 요청을 받아서 응용으로 넘긴다
	- 응용: 도메인 객체들을 조율해서 흐름을 만든다
	- 도메인: 비즈니스 규칙을 소유한다 (도메인 모델이 여기 산다)
	- 인프라: DB저장, 외부통신 등 기술적인 것만 처리한다
- **핵심**
	- 도메인 레이어는 다른 레이어를 절대 의존하지 않는다.
	- DB가 MySQL이든 MongoDB든 도메인 모델은 몰라도 된다. 
	- 이게 깨지는 순간 도메인 모델이 오염된다.
### 질문 
> Q.저 레이어들이 실제 구현레벨에서는 뭐야? 디렉터리야? 클래스야?
> A.디렉터리로 레이어를 나누고, 그 안에 클래스를 놓는다.
- 디렉터리는 경계, 클래스는 그 안의 구현체다. 
- 레이어 = 디렉터리, 역할 = 클래스라고 보면 된다.
~~~
shop/
├── presentation/          ← 표현 레이어
│   └── order_router.py    (FastAPI 라우터 클래스)
│
├── application/           ← 응용 레이어
│   └── order_service.py   (UseCase 클래스)
│
├── domain/                ← 도메인 레이어
│   ├── order.py           (Order Entity 클래스)
│   ├── order_item.py      (OrderItem Entity 클래스)
│   └── money.py           (Money Value Object 클래스)
│
└── infrastructure/        ← 인프라 레이어

- 요청 하나가 레이어를 타고 흐르는 흐름

- 도메인 모델을 기반으로 코드작성 한다
- '배송관련 규칙'에 대한 기능은 'Order 클래스'에 구현함
- 도메인의 규칙은 해당 도메인 모델(클래스)에 위치하므로,

# 1.5 도메인 모델 도출

## (1) 작업 순서 
- 도메인 모델 도출과정은 **요구사항 → 도메인 언어 추출 → 모델링 → 코드** 순서다.

### 전체 흐름

```
요구사항 (자연어)
     │
     ▼
도메인 언어 추출 (명사/동사 분리)
     │
     ▼

     │
     ▼

     │
     ▼

### 단계별 Task
  
**1단계: 요구사항 (자연어)**
- 기획자가 이렇게 던진다:
```
"고객이 상품을 장바구니에 담고 주문하면,

 주문이 확정되면 재고를 차감하고 배송을 시작한다.
 고객은 배송 시작 전까지 주문을 취소할 수 있다."
```


**2단계: 명사/동사 분리**
> 명사는 객체가 되고, 동사는 그 객체의 메서드가 된다. 
> 비즈니스 규칙은 동사에 숨어있다.

- 요구사항에서 명사와 동사를 뽑아낸다.

 고객은 배송 시작 전까지 주문을 취소할 수 있다."
```

```
class OrderId:
    def __init__(self, value: str):
        self.value = value

class OrderItem:
    def __init__(self, product_id: str, quantity: int, price: int):
        self.product_id = product_id
        self.quantity = quantity
        self.price = price

class Order:                                    # ← Aggregate Root
    def __init__(self, customer_id: str):
        self.id = OrderId(str(uuid.uuid4()))    # ← 고유 ID 생성
        self.customer_id = customer_id
        self.items = []
        self.status = "PENDING"

    def add_item(self, product_id, quantity, price):
        if self.status != "PENDING":
            raise Exception("이미 처리된 주문입니다")   # ← 비즈니스 규칙
        self.items.append(OrderItem(product_id, quantity, price))

    def confirm(self):
        if not self.items:
            raise Exception("주문 항목이 없습니다")     # ← 비즈니스 규칙
        self.status = "CONFIRMED"
        return OrderConfirmed(self.id)                  # ← Domain Event 발행

    def cancel(self):
        if self.status != "CONFIRMED":
            raise Exception("취소 불가능한 상태입니다") # ← 비즈니스 규칙
        self.status = "CANCELLED"
```

class OrderService:                    # ← 흐름만 조율, 비즈니스 규칙 없음
    def __init__(self, order_repo, event_bus):
        self.order_repo = order_repo
        self.event_bus = event_bus

    def place_order(self, customer_id, items):
        order = Order(customer_id)
        for item in items:
            order.add_item(item.product_id, item.quantity, item.price)
        
        event = order.confirm()                # ← 도메인 규칙 실행

# 1.6 엔티티와 Value
- 도메인모델 구성요소(리마인드)
~~~
1. Entity — 고유 식별자가 있는 객체 (주문, 고객)
2. Value Object — 식별자 없이 값 자체가 의미인 객체 (금액, 주소)
3. Aggregate — Entity/VO의 묶음, 일관성 경계
4. Domain Service — 객체에 속하기 애매한 비즈니스 로직
5. Repository — Aggregate를 저장/조회하는 인터페이스
6. Domain Event — 도메인에서 발생한 사건
~~~


## (1)엔티티
### 정의
Entity는 ID로 구별되는 객체다.


### 배경
DB 중심 설계에서는 모든 데이터가 테이블 row였다. 
row는 그냥 데이터 덩어리라 "이 주문과 저 주문이 같은 주문이냐"를 표현할 방법이 없었다. 
Entity는 이 문제를 해결하기 위해 나왔다.


- **가장 단순한 예시**
	- 주문 두 개가 있다.
```
주문 A
- 주문번호: 001
- 상태: 결제대기

주문 B
- 주문번호: 001
- 상태: 결제완료
```
- 같은 주문이냐 다른 주문이냐?
- **같은 주문이다. 주문번호(ID)가 같기 때문이다.** 
- 상태가 바뀌었을 뿐, 동일한 주문이 변한 것이다.

>이게 Entity의 전부다. **ID가 같으면 같은 객체, 다르면 다른 객체.**


### Entity의 특징
**1. ID가 있다**
```
┌─────────────────────────────────┐
│  Order                          │
│  id: "order-001"  ← 반드시 있음 │
│  status: PENDING                │
│  items: [...]                   │
└─────────────────────────────────┘
```

**2. 상태가 바뀐다 (mutable)**
```
┌─────────────────────────────────┐
│  주문 생성        status: PENDING    │
│       │                         │
│       ▼                         │
│  주문 확정        status: CONFIRMED  │
│       │                         │
│       ▼                         │
│  배송 시작        status: SHIPPED    │
│                                 │
│  ID는 계속 "order-001"          │
└─────────────────────────────────┘
```

**3. 비즈니스 규칙을 직접 소유한다**
```
┌─────────────────────────────────┐
│  Order                          │
│                                 │
│  + confirm()  "항목 없으면 불가" │
│  + cancel()   "배송 후 불가"    │
│  + add_item() "확정 후 추가불가" │
│                                 │
│  규칙이 객체 안에 있다           │
└─────────────────────────────────┘
```


- **코드**
```
class Order:
    def __init__(self, customer_id: str):
        self.id = str(uuid.uuid4())    # ← ID 생성, 핵심
        self.customer_id = customer_id
        self.items = []
        self.status = "PENDING"

    def confirm(self):
        if not self.items:
            raise Exception("주문 항목이 없습니다")  # ← 비즈니스 규칙
        self.status = "CONFIRMED"

    def cancel(self):
        if self.status != "CONFIRMED":
            raise Exception("취소 불가능한 상태입니다")  # ← 비즈니스 규칙
        self.status = "CANCELLED"
```



- **핵심 코드는 여기다**
	- 이 한 줄이 이 객체를 Entity로 만드는 것이다. 
	- ID가 생기는 순간 이 주문은 세상에 하나뿐인 주문이 된다.
~~~
self.id = str(uuid.uuid4())
~~~



## (2) Value 객체
### a.정의
**Value Object는 값 자체가 의미인 객체다.**


### b.왜 나왔나
금액을 그냥 `int`로 쓰면 문제가 생긴다.
- 1000이 원화인지 달러인지 모른다
- 음수 금액도 허용된다
- 금액 관련 규칙이 여기저기 흩어진다
이 문제를 해결하기 위해 값을 객체로 감싸기 시작했다.


### c.Entity와 핵심 차이

- Entity: ID로 구별한다, 상태가 바뀐다
- Value Object: 값으로 구별한다, 한번 만들면 안 바뀐다 (immutable)
1000원짜리 Money 두 개는 그냥 같다. ID가 필요 없다.

> Q.엔티티와 대등한 클래스로 구현되는 존재라고 보면 되나?
> A. 맞아. Entity도 클래스, Value Object도 클래스다. 차이는 구현 방식이다.

- Entity: ID 필드 있음, 상태 변경 메서드 있음
- Value Object: ID 없음, 값 비교(`__eq__`), 변경 시 새 객체 반환
- 둘 다 `domain/` 디렉터리 안에 나란히 존재한다.
```
domain/
├── order.py      ← Entity
├── order_item.py ← Entity
└── money.py      ← Value Object
```

### d.코드
```
class Money:
    def __init__(self, amount: int, currency: str = "KRW"):
        if amount < 0:
            raise Exception("금액은 음수가 될 수 없습니다")  # ← 비즈니스 규칙
        self.amount = amount
        self.currency = currency

    def add(self, other: "Money") -> "Money":
        if self.currency != other.currency:
            raise Exception("통화가 다릅니다")              # ← 비즈니스 규칙
        return Money(self.amount + other.amount, self.currency)  
												        # ← 새 객체 반환

    def __eq__(self, other):
        return self.amount == other.amount and self.currency == \
											        other.currency
```


- **핵심 코드 두 줄**
```
return Money(self.amount + other.amount, self.currency)
```


- 기존 객체를 바꾸지 않고 새 객체를 반환한다. 이게 immutable의 핵심이다.
	- ID 비교가 아니라 값 비교다. 이게 Value Object를 Entity와 구별하는 핵심이다.
```
def __eq__(self, other):
    return self.amount == other.amount and self.currency == other.currency
```
